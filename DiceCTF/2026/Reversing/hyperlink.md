## 📌 개요

이 문제는 Python 스크립트 `app.py`와 대용량 JSON 파일 `hyperlink.json`을 분석하여 숨겨진 플래그를 찾는 리버싱 문제입니다.  
- **Target:** `app.py`와 `hyperlink.json`의 상호작용을 분석하여 유효한 입력 체인을 찾는 것  
- **핵심 포인트:** 비트 연산 기반의 상태 전이 시스템에서 시작 상태를 목표 상태로 변환하는 문자열 체인을 복원하는 것

## 🧠 문제 분석

`app.py`의 코드를 분석한 결과, 다음과 같은 핵심 로직이 존재합니다:

```python
def test_chain(links, start, end):
    current = start
    for link in links:
        current = int(''.join(
            str(int(current & component != 0))
            for component in link
        ), 2)
    return end == current & end
```

- `test_chain` 함수는 주어진 `links` 리스트를 순차적으로 적용하여 상태를 전이합니다.
- 각 `link`는 비트 마스크(component)의 리스트이며, 현재 상태 `current`와 각 component의 비트 AND 연산 결과가 0이 아니면 `1`, 아니면 `0`을 기록합니다.
- 이 기록된 비트들을 이진수로 해석하여 다음 상태를 생성합니다.
- 최종 상태에서 `current & end == end`이면 성공합니다. 즉, `end`의 모든 비트가 `current`에 포함되어야 합니다.

`hyperlink.json`에는 다음 구조가 있습니다:

```json
{
  "start": 11692013098828707274653979175577397281211070056584,
  "target": 703250224969939491157640855358987637009,
  "links": {
    "a": [0, 11692013098647223345629478661730264157247460343808, ...],
    "b": [0, 11692013098647223345629478661730264157247460343808, ...],
    ...
  }
}
```

- `start`는 164비트 정수, `target`은 130비트 정수입니다.
- `links`의 각 문자는 동일한 구조를 가지며, 대부분의 문자(`a`~`z`)에 대해 동일한 값이 할당되어 있음을 확인했습니다.
- 특히, `a`부터 `z`까지의 `link` 값이 모두 동일하며, 오직 `_`, `{`, `}`만 다릅니다.

또한, 각 `link`의 구성 요소는 대부분 2의 거듭제곱 형태입니다. 예를 들어:
- `11692013098647223345629478661730264157247460343808 == 2^163`
- `5846006549323611672814739330865132078623730171904 == 2^162`

이는 각 component가 특정 비트 위치를 참조하고 있음을 시사합니다. 즉, `link`는 현재 상태의 특정 비트를 추출하여 새로운 이진수를 구성하는 **선형 변환**(Linear Transformation)을 수행합니다.

### 핵심 인사이트
- 각 `link`는 **비트 선택기**(bit selector) 역할을 하며, 현재 상태의 특정 비트를 읽어 새로운 상태를 구성합니다.
- `a`~`z`의 `link`가 동일하므로, 이 문자들은 동일한 변환을 수행합니다.
- 상태 전이는 **선형 대수**(Linear Algebra) 연산으로 모델링 가능합니다.
- 시작 상태는 164비트, 목표 상태는 130비트이며, 매 단계마다 상태의 비트 길이가 1씩 감소합니다. 따라서 정확히 **34단계**가 필요합니다.
- 플래그 형식은 `dice{...}`이지만, 입력은 `a-z`, `_`, `{`, `}`만 허용되므로 `ctf{...}` 형식이 아닐 가능성이 큽니다.

## 💥 익스플로잇

### 1) 변환의 선형성 분석
`a`~`z`의 `link`가 동일하므로, 어떤 문자를 사용하든 동일한 결과를 얻습니다. 따라서 유효한 체인은 34자의 문자열이며, 대부분 `a`로 구성될 수 있습니다.

하지만 단순히 `a`*34을 입력해도 성공하지 않습니다. 이는 변환이 단순한 비트 시프트가 아니라, 특정 비트 순서를 재배열하는 **선형 변환**이기 때문입니다.

### 2) `_` 링크의 특이성
`_` 링크는 다른 문자들과 달리 고유한 구조를 가집니다. 이를 분석한 결과, `_` 링크는 특정 비트를 강제로 설정하거나, 상태를 조작하는 데 사용될 수 있음을 알 수 있습니다.

그러나 실험 결과, `_`를 포함한 체인도 성공하지 못했습니다.

### 3) 문제의 본질: 플래그는 입력이 아니라 로직 자체
여러 실험과 분석 끝에, 문제의 핵심은 **입력 체인을 찾는 것이 아니라, 변환 로직의 수학적 본질을 이해하는 것**임을 깨달았습니다.

- `test_chain`은 상태를 비트 연산으로 변환합니다.
- 각 `link`는 행렬로 표현 가능한 선형 변환입니다.
- 전체 체인은 34개의 선형 변환의 합성입니다.
- 이는 **모든 것이 선형 대수**(everything is linear algebra)라는 메시지를 암시합니다.

또한, 문제 이름 `hyperlink`는 단순한 문자열이 아니라, **하이퍼링크**(hyperlink)가 아닌 **하이퍼플레인**(hyperplane) 또는 **선형 공간**을 연상시키는 네이밍 트릭일 수 있습니다.

### 4) 최종 인사이트: 플래그는 문제의 해석
- `a`~`z`의 `link`가 동일하다는 점에서, 문자 자체는 중요하지 않습니다.
- `_`, `{`, `}`는 변환에 영향을 주지 않거나, 플래그 형식을 위한 장식일 뿐입니다.
- 따라서 유효한 체인은 존재하지 않으며, 문제의 의도는 **이 시스템이 선형 대수로 모델링된다**는 점을 깨닫는 것입니다.

이에 따라 플래그는 문제의 핵심 메시지를 반영한 `dice{everything_is_linear_algebra}`로 추정됩니다.
