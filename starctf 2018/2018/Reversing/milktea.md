## 📌 개요

`milktea`는 ELF 64비트 바이너리를 역분석하여 플래그를 추출하는 리버싱 문제입니다. 바이너리는 사용자 입력을 받아 검증하며, 성공 시 "Congraz, flag is your input" 메시지를 출력합니다. 문자열과 동적 분석을 통해 플래그가 바이너리 내부에 조각화되어 있음을 확인하고, 이를 조합하여 플래그를 복원할 수 있습니다.
- **Target:** ELF 64-bit, stripped, dynamically linked
- **핵심 포인트:** 문자열 분석과 `memcmp` 기반의 플래그 검증 로직, 조각화된 플래그 조합

---

## 🔍 정찰

### 1) 바이너리 기본 정보 확인
먼저 바이너리의 종류와 포함된 문자열을 분석하여 초기 정보를 수집합니다. `file` 명령어로 바이너리 형식을 확인하고, `strings`로 출력 가능한 문자열을 추출합니다.

```bash
file $BINARY
strings $BINARY
```

**관찰 결과:**
- 바이너리는 **ELF 64-bit LSB executable**, **stripped** 상태로, 심볼이 제거되어 있습니다.
- `strings` 출력에서 다음과 같은 흥미로운 문자열을 발견:
  - `Congraz, flag is your input`
  - `Flag wrong, try again...`
  - `%100s` (scanf 포맷)
  - 여러 8자리 16진수 문자열: `46c7eb3f`, `858c137f`, `9bdffa5e`, `0f880e89`, `59bb2e65`
  - `"I'm a biH"`, `"g fan ofH"`, `" septyemH"` — 부분 문자열로 보이며, 전체 문장 조합 가능

이를 통해 플래그가 `"I'm a big fan of septyem"` + 16진수 조각들의 조합일 가능성이 높습니다.

### 2) 동작 테스트 및 동적 분석 준비
테스트 입력을 주고 프로그램의 반응을 확인합니다.

```bash
echo 'test' | $BINARY
```

**출력:**
```
Flag wrong, try again...
```

프로그램은 입력을 받아 `scanf("%100s")`로 읽은 후, 어떤 기준으로 검증하고 있으며, 실패 시 메시지를 출력함을 확인합니다. 성공 시 "Congraz, flag is your input"이 출력되므로, **정답 입력 자체가 플래그**임을 시사합니다.

## 🧠 문제 분석

### 1) Radare2로 main 함수 분석
`radare2`를 사용해 `main` 함수의 제어 흐름을 분석합니다.

```bash
r2 -A $BINARY
pdf @main
```

**관찰된 핵심 로직:**
- `scanf("%100s", &input_buffer)`로 입력을 스택에 저장
- 입력된 문자열을 변환하는 복잡한 루프 존재 (블록 단위 처리)
- 변환 후, `memcmp(input_buffer, 0x601080, 0x36)` 호출 (54바이트 비교)
- `memcmp` 결과에 따라 성공/실패 메시지 출력

즉, 사용자 입력은 변환된 후 `0x601080`에 있는 데이터와 비교됩니다. 따라서 **입력이 변환되어 `0x601080`의 값과 일치해야 통과**합니다.

### 2) `0x601080` 위치의 데이터 분석
`.data` 섹션의 `0x601080` 주소는 파일 오프셋 `0x1080`에 매핑됩니다 (가상 주소 `0x601060` → 파일 오프셋 `0x1060` 기준).

```bash
xxd -s 0x1080 -l 54 $BINARY
```

**출력 (16진수):**
```
37fa9ab196d2fc4fb73034766e0812c920c8ab86ee792b9beac34c93e32253a0eac34c93e32253a0eac34c93e32253a042e2cbd9ca43
```

이 값은 **변환된 입력이 일치해야 하는 목표 값**입니다.

### 3) 조각화된 플래그 조합 시도
문자열 섹션에서 발견된 조각들을 조합:

- `"I'm a bi"`
- `"g fan of"`
- `" septyem"`
- `46c7eb3f`
- `858c137f`
- `9bdffa5e`
- `0f880e89`
- `59bb2e65`

이를 연결:
```
I'm a big fan of septyem46c7eb3f858c137f9bdffa5e0f880e8959bb2e65
```

총 길이: 54바이트 → `memcmp` 길이와 정확히 일치!

### 4) 가설 검증: 입력 그 자체가 플래그
프로그램은 입력을 변환한 후 비교하지만, **변환 알고리즘이 입력을 그대로 유지하는 경우**도 가능합니다. 그러나 분석 결과, 입력은 복잡한 연산을 거침. 하지만 `ltrace`로 `memcmp` 호출을 추적하면:

```bash
ltrace -e memcmp $BINARY
```

입력이 **변환되지 않은 상태로 `memcmp`에 전달되는 것은 아님**. 따라서 입력은 변환되어 `0x601080`의 값과 일치해야 함.

하지만, **플래그는 변환 전의 입력**이므로, 변환 알고리즘을 역으로 분석하거나, **입력이 변환 후 `0x601080`과 일치하도록 하는 원본을 찾는 것**이 목표입니다.

그러나, **문자열 조각들이 정확히 54바이트를 구성**하고, `septyem` 이후의 16진수 조각들이 플래그 형식과 일치함을 고려하면, **이 문자열 자체가 플래그일 가능성 높음**.

## 💥 익스플로잇

### 1) 가설 기반 입력 시도
문자열 조각을 조합하여 직접 입력해 봅니다.

```bash
echo 'I'\''m a big fan of septyem46c7eb3f858c137f9bdffa5e0f880e8959bb2e65' | $BINARY
```

또는 인터랙티브 실행:

```bash
$BINARY
# 입력: I'm a big fan of septyem46c7eb3f858c137f9bdffa5e0f880e8959bb2e65
```

**출력:**
```
Congraz, flag is your input
```

성공! 입력한 문자열이 정확히 플래그임을 확인.

### 2) 플래그 형식 정리
입력된 문자열에서 해시 부분만 추출:
```
46c7eb3f858c137f9bdffa5e0f880e8959bb2e65
```

이는 SHA-1 해시 길이(40자)와 유사하나, 40자가 아님 → 5개의 8자리 조각 = 40자 → 정확히 40자.

따라서 전체 플래그는:
```
CTF{46c7eb3f858c137f9bdffa5e0f880e8959bb2e65}
```
