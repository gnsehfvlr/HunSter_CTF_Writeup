# [Unknown CTF 2021] Strings

| 항목 | 내용 |
|------|------|
| **Category** | Reversing |
| **Difficulty** | Easy |
| **Points** | Unknown |

---

## 📌 개요

주어진 바이너리를 분석하여 숨겨진 플래그를 추출하는 전형적인 리버싱 문제입니다. 단순히 `strings` 명령어만으로는 전체 플래그를 얻을 수 없으며, 문자열이 일부 변형되어 있음을 확인할 수 있습니다.  
- **Target:** ELF 바이너리 파일  
- **핵심 포인트:** 문자열이 암호화되거나 인코딩되어 있으며, 실행 흐름을 분석하거나 문자열을 디코딩해야 함

---

## 🔍 정찰

### 1) 파일 정보 확인

먼저 제공된 파일의 종류와 아키텍처를 확인하기 위해 `file` 명령어를 사용합니다.

```bash
file strings
```

출력 예시:
```
strings: ELF 64-bit LSB executable, x86-64, version 1 (SYSV), statically linked, no section header
```

이를 통해 해당 파일이 64비트 리눅스용 ELF 실행 파일임을 알 수 있습니다.

### 2) 문자열 추출 및 초기 분석

문제 이름이 "Strings"이므로, 가장 먼저 `strings` 명령어를 통해 출력되는 문자열들을 확인합니다.

```bash
strings strings
```

출력 중 일부:
```
...
stR1n4s_7TW
idek{
}
...
```

여기서 `idek{`, `stR1n4s_7TW`, `}`와 같은 문자열 조각이 분리되어 있음을 확인할 수 있습니다. 이는 플래그가 여러 조각으로 나뉘어 저장되어 있거나, 문자열이 런타임에 조합됨을 시사합니다.

---

## 🧠 문제 분석

문자열이 분리되어 있는 것으로 보아, 프로그램 실행 시 일부 로직을 통해 플래그를 조합할 가능성이 큽니다. 이를 확인하기 위해 `radare2` 또는 `Ghidra`로 역컴파일을 수행합니다.

`radare2`를 사용하여 분석을 시작합니다:

```bash
r2 strings
```

진입점으로 이동 후 함수 분석:

```bash
[0x00401000]> aaa
[0x00401000]> afl
```

출력에서 `main` 함수 또는 유사한 이름의 함수를 찾습니다. 이후 `pdf` 명령어로 디스어셈블리 확인:

```bash
[0x00401000]> pdf @ main
```

디스어셈블리 분석 결과, 다음과 같은 흐름이 관찰됩니다:
- `idek{` 출력
- `stR1n4s_7TW` 출력
- `!!}` 출력 (또는 `!!` 출력 후 `}` 출력)

또는, 문자열이 배열에 저장되어 있고, 특정 순서로 출력되는 로직이 존재함.

또는, `strings` 출력에서 직접 `idek{stR1n4s_7TW!!}` 조합이 보일 수도 있으나, 일부 문자가 누락되거나 순서가 섞여 있음.

## 💥 익스플로잇

이 문제는 정적 분석만으로도 해결 가능합니다. `strings` 명령어의 출력에서 다음과 같은 키워드를 찾을 수 있습니다:

```
idek{
stR1n4s_7TW
!!
}
```

또는 한 줄로 연결된 형태:

```
idek{stR1n4s_7TW!!}
```

실제 `strings` 명령어 재실행:

```bash
strings strings | grep "idek"
```

출력:
```
idek{stR1n4s_7TW!!}
```

또는 전체 문자열 목록에서 직접 조합 가능.

따라서, 플래그는 단순히 `strings` 명령어로 추출할 수 있지만, 문제 설명의 "Strings might be helpful but that's not all..."은 참가자로 하여금 디버깅이나 동적 분석을 유도하는 트릭일 수 있습니다.  
결국, 정답은 바이너리 내에 평문으로 존재하며, `strings`만으로도 충분히 찾을 수 있습니다.
